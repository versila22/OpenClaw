# 2026-03-22 : Industrialisation du projet "Régisseur IA" (DJ_Impro) et Debugging Système

## Axe 1 : Projet "DJ_Impro" - de l'outil à l'écosystème Open Source

### Initialisation et Isolation
- **Objectif :** Isoler la PWA de télécommande d'improvisation pour la publier sur GitHub.
- **Actions :**
    - Création d'un dossier de projet dédié : `impro-remote/`.
    - Refactoring du code pour éliminer les chemins en dur et le rendre portable.
    - Création de l'infrastructure de base : `README.md`, `requirements.txt`, et un `.gitignore` strict pour protéger les secrets (`.env`, `.cache`).
- **Décision :** Le projet est nommé `DJ_Impro` et sera publié sur le compte GitHub `versila22`.

### Infrastructure Communautaire (Axe Horizontal)
- **Objectif :** Rendre le projet "prêt à contribuer" pour la communauté des "geeks improvisateurs".
- **Actions :**
    - Création des documents de gouvernance : `CONTRIBUTING.md` et `ARCHITECTURE.md`.
    - Mise en place de templates d'Issues GitHub pour les bugs et les demandes de fonctionnalités.
    - Création d'un système de gamification via `BOUNTIES.md` pour encourager le développement de plugins demandés (QLab, DMX, Apple Music).
- **Décision :** Le projet est marketé comme "Le premier Régisseur IA Open Source pour le spectacle vivant" pour attirer les contributeurs techniques.

### Architecture Plugins
- **Objectif :** Passer d'un script monolithique à une architecture modulaire.
- **Actions :**
    - Le serveur Flask (`impro_remote_server.py`) a été réécrit pour devenir un "Core" agnostique.
    - Le Core charge dynamiquement des plugins depuis un dossier `plugins/`.
    - L'interface PWA est générée dynamiquement en fonction des configurations des plugins chargés.
    - La logique Spotify a été extraite dans `plugins/spotify_plugin.py`.

## Axe 2 : Vision IA (Axe Vertical)
- **Objectif :** Transformer la télécommande en un agent IA autonome.
- **Roadmap définie :**
    1. **Ingestion :** Créer des outils pour télécharger des vidéos YouTube d'impro et les transcrire avec Whisper.
    2. **Décision :** Utiliser un LLM pour analyser les transcripts et générer un JSON de "timing musical".
    3. **Audit :** Créer des outils (via `.srt` ou `FFmpeg`) pour visualiser et valider les décisions de l'IA sur les vidéos de test.
    4. **Live :** Adapter l'architecture pour une analyse en temps réel avec une faible latence.

## Axe 3 : Debugging & Stabilisation Système
- **Problème :** Multiples blocages lors des tests et pannes système.
- **Actions de résolution :**
    - **Déploiement GitHub :** Mise en place d'une Deploy Key SSH dédiée au projet `DJ_Impro` pour me donner un accès autonome en écriture.
    - **Test de la PWA :** Diagnostic et résolution de problèmes réseau complexes (ports Docker non mappés, tunnels SSH, authentification `localtunnel`). Correction d'un bug de "serveur fantôme" bloquant le port 5000 avec `fuser -k`.
    - **Stabilité OpenClaw :**
        - Correction d'un crash du Gateway Telegram causé par des permissions de fichier incorrectes sur `openclaw.json` (propriétaire `root` au lieu de `node`).
        - Débogage et correction de la configuration des modèles LLM. Remplacement des modèles "preview" instables par des versions de production stables.
- **Décision finale :** Le modèle par défaut du système est maintenant `google/gemini-2.5-flash` pour un équilibre optimal entre coût, vitesse et performance. Les autres modèles stables (`gemini-2.5-pro`, `gemini-pro`) sont disponibles via des alias pour des tâches spécifiques.
